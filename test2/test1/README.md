# 🏭 ИИ-ассистент завода — Чат-бот по базе данных

ИИ-ассистент с веб-интерфейсом, который отвечает на вопросы о производстве, оборудовании и персонале. Принимает вопрос на естественном языке, генерирует SQL-запрос к базе данных и возвращает результат в понятном виде.

## Стек технологий

- **Python 3.10+**
- **Django 5.x** — веб-фреймворк
- **Ollama (qwen3:14b)** — локальная LLM для Text-to-SQL
- **SQLite** — производственная база данных

## Архитектура

```
┌─────────────────────────────────────────────────────────┐
│                    Веб-интерфейс                        │
│              (HTML/CSS/JS — чат)                        │
├─────────────────────────────────────────────────────────┤
│                     Django Views                        │
│              (обработка HTTP/API)                       │
├────────────┬──────────────────┬─────────────────────────┤
│  Text-to-  │  DB Executor     │  Answer                 │
│   SQL      │  (SQLite)        │  Generator              │
│  (Ollama)  │                  │  (Ollama)               │
├────────────┴──────────────────┴─────────────────────────┤
│              Семантический слой                         │
│         (бизнес-термины → таблицы БД)                   │
├─────────────────────────────────────────────────────────┤
│              Factory Database (SQLite)                  │
│    mdc_*  hr_*  rep_*  tp_*  (40+ таблиц)              │
└─────────────────────────────────────────────────────────┘
```

## Как это работает

1. **Пользователь** задаёт вопрос на русском языке
2. **Text-to-SQL модуль** (через Ollama qwen3:14b) генерирует SQL-запрос
3. **DB Executor** выполняет запрос к SQLite
4. **Answer Generator** (через Ollama) формирует понятный ответ с таблицами
5. Результат отображается в чат-интерфейсе

## Установка и запуск

### Предварительные требования

1. **Python 3.10+**
2. **Ollama** — должен быть установлен и запущен
3. **Модель qwen3:14b** — загружена в Ollama

### Запуск Ollama

```bash
# Запуск Ollama сервера
ollama serve

# Проверка (в отдельном терминале)
ollama list
# Должна быть видна: qwen3:14b
```

### Запуск приложения

```bash
# Установка зависимостей
pip install -r requirements.txt

# Миграции Django
python manage.py migrate

# Запуск сервера
python manage.py runserver 8000
```

Откройте браузер: **http://localhost:8000**

## API Endpoints

| Метод | URL | Описание |
|-------|-----|----------|
| GET | `/` | Главная страница чата |
| POST | `/api/chat/` | Отправить сообщение в чат |
| POST | `/api/check/` | Проверить подключение к Ollama |
| GET | `/api/session/` | Информация о сессии |
| POST | `/api/reset/` | Сбросить сессию |

### Формат запроса к `/api/chat/`

```json
{
    "message": "Какие станки есть в цехе 5?",
    "session_id": "unique_session_id"
}
```

### Формат ответа

```json
{
    "answer": "Текст ответа на русском языке",
    "sql": "SELECT ...",
    "data": {
        "columns": ["col1", "col2"],
        "rows": [{"col1": "val1", "col2": "val2"}],
        "count": 3
    },
    "error": null
}
```

## Подход: Text-to-SQL + Семантический слой

Вместо RAG используется **прямой Text-to-SQL** с семантическим слоем:

- **Системный промпт** содержит полную схему БД и связи таблиц
- **Семантический словарь** маппит бизнес-термины на таблицы/столбцы
- **Few-shot примеры** показывают LLM правильные паттерны запросов
- **Валидация** проверяет безопасность (только SELECT)

### Почему не RAG?

RAG оптимален для неструктурированных документов. Для SQL-баз данных:
- Эмбеддинги не помогут сгенерировать корректный JOIN
- Векторный поиск не понимает структуру таблиц
- Прямой Text-to-SQL точнее и быстрее

## Безопасность

- Валидация SQL на опасные команды (DROP, DELETE, INSERT...)
- Только SELECT-запросы разрешены
- Таймауты на запросы к Ollama и БД
- CSRF-защита в Django

В `zavod_assistant/settings.py`:

```python
# Путь к производственной БД

OLLAMA_BASE_URL = 'http://localhost:11434'

```
